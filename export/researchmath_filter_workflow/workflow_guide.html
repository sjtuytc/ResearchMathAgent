1. Overview

The ResearchMath dataset contains approximately 22,000 open mathematical research problems drawn from arXiv papers, the Erdős problem list, AIM workshops, Google DeepMind's Formal Conjectures, and an unsolved problems compendium. Problems span analysis, number theory, algebraic combinatorics, topology, graph theory, and eight other domains.

The dataset was constructed by an agentic pipeline (see arXiv:2605.28003) that extracted problems from source documents and verified their open/solved status via literature search. However, no systematic human filtering or AI-tractability ranking exists. This workflow fills that gap.

Goals of this workflow

  • Score each problem on five dimensions relevant to AI-assisted research
  • Produce a ranked shortlist of problems worth prioritising for deep work
  • Enable distributed annotation: different people can score different datasets

What "filtering" and "ranking" mean here

Filtering is a binary/ternary decision: is this problem worth deeper attention? A problem may be excluded because it is poorly stated, extremely niche, or requires experimental/physical knowledge that AI cannot access.

Ranking orders the included problems by a weighted score across five dimensions. This score reflects how good a fit the problem is for AI-assisted mathematical investigation — not how easy it is to solve.

2. Quick Start

1
Install the dependency 
pip install anthropic
2
Set your Anthropic API key 
export ANTHROPIC_API_KEY="sk-ant-..."
3
Run the scorer 
# Score 50 problems from the main dataset
python rank_problems.py --dataset researchmath_14k --limit 50

# Score all datasets, 20 problems each (faster model = cheaper)
python rank_problems.py --all-datasets --limit 20 \
    --model claude-haiku-4-5-20251001
4
Review the output
Results appear in ranked_output/ as JSON and CSV files, sorted from highest to lowest rank score.
API cost estimate. Scoring 50 problems with claude-opus-4-8 costs roughly $0.05–$0.15 depending on problem length. Using claude-haiku-4-5-20251001 reduces cost by ~20×. See Appendix A for a cost table.

4. Scoring Rubric

Each problem is scored on five dimensions, each from 1 (lowest) to 5 (highest). The final rank score is a weighted average:

rank_score = 0.30 × importance
           + 0.25 × ai_tractability
           + 0.20 × verifiability
           + 0.15 × clarity
           + 0.10 × self_containedness

Dimension Definitions

Importance (weight 30%)

Mathematical or scientific significance of solving this problem. How many researchers would care? Does it resolve a longstanding question or unlock a new area?

1Recreational or very niche curiosity
2Of interest within one narrow subfield
3Recognized open problem in its area
4Significant; would attract broad attention if solved
5Central open problem (e.g. Riemann Hypothesis-class)
AI Tractability (weight 25%)

How amenable is this to AI-assisted research? Does it benefit from exhaustive search, generating many candidate approaches, or formal verification?

1Requires deep human geometric intuition or physical experiment
2Mostly human insight; AI can only assist marginally
3AI can meaningfully explore sub-cases or related lemmas
4Strong candidate for AI: verifiable steps, structured search space
5Amenable to formal proof search or systematic case enumeration
Verifiability (weight 20%)

Can a proposed solution be checked rigorously? Are success criteria concrete?

1No clear criterion; "improvement" is subjective
2Solution could be checked but requires extensive expertise
3Checkable with effort by a domain expert
4Clear logical proof or counterexample verifiable by peer review
5Machine-checkable (formal proof assistant, computer search)
Clarity (weight 15%)

Is the problem precisely stated? Are all terms defined? Is it unambiguous what counts as a solution?

1Vague, undefined terms, or question is ambiguous
3Mostly clear but requires background to interpret fully
5Formally stated; a logician could parse it immediately
Self-Containedness (weight 10%)

Can the problem be understood from its statement alone, without consulting specific papers?

1Completely depends on a specific theorem/definition from a paper
3Requires standard graduate-level background but no specific paper
5Fully self-contained; readable by any mathematician

Recommendations

Based on the rank score, Claude gives a recommendation:

  • include — Rank score ≥ 3.5; good candidate for deep work
  • borderline — Rank score 2.5–3.5; worth human review
  • exclude — Rank score < 2.5; likely not a good fit
Tip: The weights and thresholds are configurable in config_example.yaml. If your use case values formal proofs more, increase verifiability. If you want problems relevant to a specific domain, pre-filter using the filter website first.

5. Using the Filter Website

The filter website lets you browse all datasets before running the script, which helps you get a sense of the data and pre-filter to a manageable scope.

URLs

  • Production: https://zipfile-legume-gaining.ngrok-free.dev/rmac/filter/
  • Insights tab: click "Insights" in the tab bar to see per-dataset statistics and auto-generated todos

Typical workflow before scoring

  1. Open the filter website and select a dataset
  2. Use the domain filter to narrow to the area you want to focus on
  3. Use the status chip to filter to open or unknown problems
  4. Note the domain name and problem count — pass these to the script via --status open

The filter website is read-only and does not modify any data.

6. Running the Scoring Script

Prerequisites

pip install anthropic          # Required
pip install pyyaml             # Optional: needed only for --config

An Anthropic API key is required. Get one at console.anthropic.com.

Common usage patterns

# ── Basic ──────────────────────────────────────────────────
# Score the first 50 problems from ResearchMath-14k
python rank_problems.py --dataset researchmath_14k --limit 50

# Score only 'open' problems (fastest subset with known status)
python rank_problems.py --dataset researchmath_14k --status open --limit 100

# Use a cheaper model for a quick first pass
python rank_problems.py --dataset erdos_problems --limit 200 \
    --model claude-haiku-4-5-20251001

# ── All datasets ────────────────────────────────────────────
# 20 problems from each of the 5 datasets
python rank_problems.py --all-datasets --limit 20

# ── Custom output dir ───────────────────────────────────────
python rank_problems.py --dataset formal_conjectures --limit 50 \
    --output results/formal_conjectures_june26

# ── Using a config file ─────────────────────────────────────
cp config_example.yaml config.yaml
# Edit config.yaml with your settings
python rank_problems.py --config config.yaml

# ── Local data (if you have the dataset files) ──────────────
python rank_problems.py --dataset researchmath_14k \
    --local-data /path/to/ResearchMathAgent-web/data/datasets \
    --limit 500

7. Understanding the Output

Each run produces a JSON file and a CSV file in the output directory:

ranked_output/
├── researchmath_14k_ranked.json   # Full results with reasoning
├── researchmath_14k_ranked.csv    # Spreadsheet-friendly view
└── all_datasets_ranked.json       # Combined (if --all-datasets)

JSON output structure (one entry)

{
  "rank": 1,
  "dataset": "researchmath_14k",
  "id": "rm14k_00042",
  "title": "Bounding the chromatic number of Kneser graphs",
  "domain": "Discrete Mathematics and Combinatorics",
  "status": "open",
  "rank_score": 4.275,
  "recommendation": "include",
  "clarity": 5,
  "importance": 4,
  "ai_tractability": 4,
  "verifiability": 5,
  "self_containedness": 4,
  "brief_reasoning": "The problem has a crisp graph-theoretic statement
    with no ambiguity. Chromatic number bounds are central to combinatorics
    and the question is well-studied enough that AI progress could be
    verified against known results."
}

Interpreting the CSV

The CSV has the same fields. Sort by rank_score descending and filter by recommendation = include to get your shortlist. Open in Excel or Google Sheets and use conditional formatting on rank_score to visualise the distribution.

8. Human Review Guidelines

Claude's scores are a starting point, not a final verdict. A human reviewer should:

  1. Spot-check high-scoring "includes" — Read the brief_reasoning and verify the score makes sense for the actual problem statement.
  2. Rescue "borderline" problems you know — If you recognise a problem as important from domain expertise, override the recommendation to "include".
  3. Reject "includes" with vague statements — If Claude gave high clarity but the statement relies on undefined notation, downgrade it.
  4. Note disagreement patterns — If Claude consistently overscores a domain (e.g. formal conjectures) you can re-weight dimensions in config.yaml.
Dividing the work. If multiple people are annotating, assign datasets or domain slices rather than problem-level splits. This ensures each person builds domain familiarity rather than reviewing unrelated problems.

Suggested review protocol

  1. Run the script with --limit 50 as a pilot to calibrate.
  2. Discuss 5–10 disagreements with collaborators to align on interpretation.
  3. Re-run with agreed weight adjustments and --limit 200+.
  4. Each reviewer independently reads the top 20% by rank_score and flags upgrades/downgrades.
  5. Merge decisions: if any reviewer includes, include; if all exclude, exclude.

D. Modifying the scoring prompt

The prompt is in rank_problems.py as the constant SCORING_PROMPT. You can change the dimension definitions, add new dimensions, or target a different research goal (e.g. "problem suitability for a graduate seminar" or "prize problem tractability"). If you add or remove dimensions, update RANK_WEIGHTS accordingly.
